Proof-Carrying TypeScript for the agent-written software era.

The product direction from this conversation is a Lean-backed verification layer for TypeScript packages: agents produce specs, code, and proofs; Lean checks the proofs; React and Bun consume ordinary generated output.

Core Bet

Do not make arbitrary source code the source of truth. Make specifications and proofs the source of truth, then generate or check executable TypeScript/JavaScript against them.

Conversation: Is it possible to write a Lean compiler that turns high level code like TypeScript into provable Lean code?

Answer: Yes, but only in a precise, constrained sense. You can write a compiler or transpiler that takes a high-level language such as TypeScript and emits Lean definitions plus proof obligations. Lean is suitable because it is both a programming language and an interactive theorem prover; its kernel checks proof terms, tactics generate proof terms that are still kernel-checked, and Lean 4 has strong metaprogramming support for syntax translators, DSLs, proof automation, and generated Lean artifacts.

The hard part is not generating Lean code. The hard part is making generated Lean code faithfully mean the same thing as the original TypeScript and then proving useful properties about it.

Realistic pipeline: TypeScript source -> TypeScript parser / compiler API -> restricted typed AST -> Lean model of the program -> generated specifications / proof obligations -> Lean proofs, partly automated and partly manual.

TypeScript's compiler API exposes AST creation, printing, and program construction facilities, so front-end parsing and extraction are feasible. But TypeScript is a static type checker for JavaScript. Its types are erased when compiled, and type annotations do not change runtime behavior. That matters because a TypeScript function such as:

function abs(n: number): number {
  return n < 0 ? -n : n
}

could become a Lean model:

def absModel (n : Int) : Int :=
  if n < 0 then -n else n

theorem absModel_nonnegative (n : Int) : 0 <= absModel n := by
  unfold absModel
  split
  . omega
  . omega

But this is not literally TypeScript semantics. TypeScript number is JavaScript number, with floating-point behavior, NaN, Infinity, -0, overflow behavior, and coercions. Translating it as Lean Int verifies an abstract model, not exact runtime behavior. That can be acceptable only if the compiler/runtime enforces or proves that abstraction is valid.

Practical feasibility: parsing TypeScript and emitting Lean syntax is straightforward compiler engineering. Emitting Lean models for a restricted TypeScript subset is very feasible. Generating theorem statements from annotations is feasible and resembles Dafny-style pre/postconditions. Automatically proving simple properties such as arithmetic, algebraic data type behavior, pure functions, and small-domain invariants is feasible. Automatically proving arbitrary app correctness is not feasible in general because the intended property must be specified and many proofs require guidance. Faithfully modeling all JavaScript/TypeScript runtime semantics is technically possible but huge because of mutation, prototypes, async, exceptions, NaN, DOM APIs, Node APIs, and more. Proving the compiler itself correct is possible but much harder because it requires formal source semantics, target semantics, and a semantic preservation theorem.

Existing analogues: Aeneas translates Rust programs into proof-assistant models, including Lean, so functional correctness can be verified. LemmaScript is a TypeScript-oriented tech preview that generates verifiable code from annotated TypeScript into Dafny or Lean 4, while its README labels support, semantics, and ergonomics as still evolving. Dafny is relevant because it is verification-aware, has specifications built into the language, and can compile verified programs to JavaScript and other targets.

The credible design should not attempt all TypeScript. It should define a verification subset. Allowed: pure functions, algebraic-style data, records/objects with fixed fields, arrays with explicit bounds reasoning, finite maps/sets, structural recursion or checked termination, and annotations for requires/ensures/invariants. Restricted or excluded initially: any/unknown without refinement, prototype mutation, eval, reflection-heavy code, implicit coercions, floating-point unless modeled explicitly, async/event-loop behavior, arbitrary DOM/Node APIs, exceptions unless modeled with Except, and mutation unless modeled with State.

Generated Lean should represent effects explicitly:

-- Pure function
def f : Input -> Output := ...

-- Possible failure
def f : Input -> Except Error Output := ...

-- Mutable state
def f : Input -> State AppState Output := ...

-- IO boundary, usually specified abstractly
def f : Input -> IO Output := ...

For a serious system, use three layers: source semantics as a formal Lean definition of the supported TypeScript subset; a translator from TypeScript AST to Lean model code; and a correctness theorem saying that for every supported source program p, evaluating p under the TypeScript-subset semantics corresponds to evaluating translate(p) in Lean. Without layer 3, generated Lean code is still useful, but the translator is in the trusted computing base. With layer 3, the project moves toward a verified compiler or verified extraction pipeline.

The key limitation: a compiler cannot magically infer what the program should do. It can generate proof obligations such as:

//@ requires n >= 0
//@ ensures result >= n
function inc(n: int): int {
  return n + 1
}

Then emit:

def inc (n : Int) : Int := n + 1

theorem inc_spec (n : Int) :
  n >= 0 -> inc n >= n := by
  intro h
  unfold inc
  omega

That is realistic, but the specification has to come from annotations, a schema, a type refinement system, generated invariants, or human-written Lean. Best MVP: TypeScript-like subset -> Lean pure functions -> generated theorems from //@ annotations. Then expand into arrays, maps, mutation, exceptions, and async. Trying to model arbitrary production TypeScript first would bury the project in JavaScript edge cases before producing useful verification value.

Conclusion: TypeScript to provable Lean is possible and adjacent projects prove the approach, but it should be framed as verified extraction for a disciplined subset, not automatic full verification of arbitrary TypeScript.

Question: Could this be compatible with React and Bun?

Answer: Yes. React and Bun compatibility is feasible, especially if the language compiles to ordinary JavaScript/TypeScript/TSX while emitting Lean for proofs.

Clean architecture: Verified TS/Lean-like source -> Lean output for proofs + TS/JS/TSX output for runtime + .d.ts output for TypeScript ecosystem compatibility -> React and Bun consume generated runtime code.

React compatibility: React is a good target because components are close to pure functions: components are JavaScript functions that return markup, and JSX is common syntax. A language could support JSX-like syntax:

type User = {
  id: UserId
  name: NonEmptyString
}

component UserCard(props: { user: User }): ReactNode
  ensures renderedText(result).contains(props.user.name)
{
  <div className="user-card">
    <h2>{props.user.name}</h2>
  </div>
}

Then generate ordinary React-compatible TSX:

export function UserCard(props: UserCardProps) {
  return (
    <div className="user-card">
      <h2>{props.user.name}</h2>
    </div>
  );
}

And separately generate Lean:

def UserCardModel (props : UserCardProps) : VNode :=
  div [className "user-card"]
    [h2 [] [text props.user.name]]

theorem UserCard_contains_name (props : UserCardProps) :
  renderedText (UserCardModel props) contains props.user.name := by
  ...

This works best if React itself is not verified initially. Verify the component model and treat React as runtime renderer. Verified: props shape, state transitions, reducer correctness, UI invariants, impossible states, form validation, routing/data constraints, pure rendering model. Not fully verified initially: React reconciler, browser DOM, CSS layout, event loop, third-party component internals, network behavior. This is still valuable because many frontend bugs come from invalid state, impossible props, unchecked data, inconsistent reducers, and unsafe assumptions, not from React's reconciler being wrong.

Bun compatibility: Bun already supports TypeScript, JSX, and TSX as runtime/build inputs. Bun would not need to know about Lean; it would run or bundle generated JavaScript/TypeScript/TSX. Possible emitted structure:

dist/
  index.js
  index.d.ts
  UserCard.js
  UserCard.d.ts
  proofs/UserCard.lean

or during development:

src/UserCard.vtsx
generated/UserCard.tsx
generated/UserCard.lean

Most practical compatibility model: a verified language layer that is proof-aware, TS-like, and JSX-capable; a JavaScript ecosystem layer with generated TSX/JS, npm package compatibility, React components, and Bun runtime; and a proof layer with Lean definitions and theorems checked by Lean. The language should distinguish verified code from external code:

extern component Button(props: ButtonProps): ReactNode
  assumes button_renders_label: renderedText(result).contains(props.label)

This allows ordinary React libraries through explicit contracts. Third-party components are imported behind assumptions or runtime checks.

Best first target: TS-like syntax, JSX support, React function components, typed props, pure components, verified reducers, generated .tsx, generated .d.ts, generated Lean, and Bun-compatible output. Example:

type CounterState = {
  count: Nat
}

type CounterAction =
  | { type: "inc" }
  | { type: "dec" }

fn reducer(state: CounterState, action: CounterAction): CounterState
  ensures result.count >= 0
{
  match action {
    { type: "inc" } => { count: state.count + 1 }
    { type: "dec" } =>
      if state.count == 0 {
        { count: 0 }
      } else {
        { count: state.count - 1 }
      }
  }
}

Generated React code can use it normally: const [state, dispatch] = useReducer(reducer, { count: 0 }); Generated Lean proves the reducer never creates a negative count. This is a strong use case: verified state machines feeding ordinary React UI.

Hard parts are not Bun or React compatibility. Hard parts are proof ergonomics, readable generated Lean, clean effects, async/network APIs, React hook modeling, contracts for external components, and avoiding the temptation to support all JavaScript behavior. Realistic claim: verified source compiles to Lean for proofs and to TSX/JS for React/Bun runtime compatibility. Unrealistic claim: the entire React+Bun+browser runtime is fully verified.

Question: If we were to make TypeScript and its entire ecosystem, including Bun and React, verifiable like Lean, what strategy or open-source project would have the most impact?

Answer: The highest-impact strategy would be proof-carrying TypeScript packages, not verifying all of TypeScript, Bun, React, and npm in one pass.

Analogy: today an npm package ships dist/index.js, dist/index.d.ts, and package.json. In the future it could ship dist/index.js, dist/index.d.ts, proofs/*.lean, contracts/*.json, specs/*.vtspec, and package.json declaring verified exports.

The project with the most impact would be an open-source Lean-backed verification layer for the TypeScript ecosystem: a tool that lets package authors write ordinary or near-ordinary TypeScript, add specs, generate Lean proof obligations, and publish proof artifacts alongside the runtime package.

Possible names: proof-ts, verified-ts, lean-ts, proofpack. Core promise: write TypeScript, add specifications, generate Lean, check proofs, and ship the same JS/TS package with machine-checkable contracts.

The goal is not to prove React, Bun, V8, JavaScriptCore, every npm package, and the browser DOM immediately. The goal is composable verification at package boundaries. TypeScript itself scaled by letting packages ship type information, not by proving all JavaScript correct. Verification should scale the same way.

Package boundaries matter because the TypeScript ecosystem is package-shaped. React apps, Bun servers, libraries, CLIs, schemas, routers, UI kits, SDKs, and database clients compose through npm packages. The scalable proof unit is exported function/component/hook/reducer/schema/API contract, not entire application + entire runtime + entire browser + all dependencies.

Example verified export:

export function transfer(
  ledger: Ledger,
  from: AccountId,
  to: AccountId,
  amount: PositiveInt
): Ledger
  requires hasAccount(ledger, from)
  requires hasAccount(ledger, to)
  requires balanceOf(ledger, from) >= amount
  ensures totalBalance(result) == totalBalance(ledger)
  ensures balanceOf(result, from) == balanceOf(ledger, from) - amount
  ensures balanceOf(result, to) == balanceOf(ledger, to) + amount

Then the package ships Lean proofs that those claims hold.

Realistic architecture: TypeScript/TSX source -> verification extractor -> small verified intermediate language TSCore -> Lean model and theorem obligations -> Lean checks proofs -> runtime JS/TS remains usable by Bun, React, Node, and browsers -> package publishes proof manifest.

Closest existing project: LemmaScript, a TypeScript verification toolchain where ordinary TypeScript is annotated with //@ specifications and generates verifiable code in Dafny or Lean 4. It is a tech preview but points in the right direction. Aeneas for Rust is an analogue because it translates Rust programs into functional proof-assistant models; the lesson is to translate a disciplined source subset into a pure model pleasant to prove against. Dafny is relevant because it already combines specs, static verification, and compilation to ordinary languages including JavaScript.

First valuable targets are not React's reconciler or Bun internals. They are bug-heavy boundaries: domain models, reducers and state machines, API request/response contracts, schema validators, authorization checks, data transformations, routing invariants, form invariants, financial/accounting logic, and protocol encoders/decoders.

For React, first target verified reducers + verified props + verified state transitions, not verified DOM rendering. For Bun, first target verified server handlers + verified route contracts + verified package build/check pipeline, not JavaScriptCore or all Bun internals.

Proof-carrying npm manifest idea:

{
  "name": "@acme/ledger",
  "version": "1.0.0",
  "main": "dist/index.js",
  "types": "dist/index.d.ts",
  "proofs": {
    "checker": "lean4",
    "manifest": "proofs/manifest.json",
    "root": "proofs/Acme/Ledger.lean"
  },
  "verifiedExports": {
    "./transfer": {
      "spec": "contracts/transfer.vtspec",
      "theorems": [
        "Acme.Ledger.transfer_preserves_total_balance",
        "Acme.Ledger.transfer_debits_sender",
        "Acme.Ledger.transfer_credits_receiver"
      ]
    }
  }
}

Consumers could run bunx proof-ts verify or npx proof-ts verify and get checks for TypeScript types, generated runtime contracts, Lean proofs, theorem manifest validity, and implementation/model correspondence. This borrows proof-carrying code: code is accompanied by formal evidence that it satisfies a policy, and the consumer checks the proof instead of trusting the producer.

Adoption ladder: Level 0 plain TypeScript; Level 1 runtime schemas for boundary data; Level 2 contracts with requires/ensures/invariant; Level 3 generated proof obligations; Level 4 verified implementation; Level 5 proof-carrying package; Level 6 verified dependency graph. This matters because TypeScript is gradual, so verification should be gradual too.

First killer app: a verified React+Bun app template called create-verified-app, with Bun runtime, React frontend, verified domain model, verified reducer, verified API schemas, generated Lean proofs, generated runtime validators, and a proof badge in CI. Example target: collaborative todo app with users, projects, task state transitions, permissions, optimistic UI updates, server route contracts, and database boundary checks. Prove archived tasks cannot be edited, users cannot mutate projects they do not belong to, completed tasks always have completion timestamps, optimistic updates preserve domain invariants, API handlers never return malformed domain objects, and reducer transitions preserve valid UI state.

Most important technical component: a small formal intermediate language, not direct arbitrary TypeScript. Call it TSCore. Include Int, Nat, Bool, String, records, tagged unions, arrays/lists, maps, pure functions, pattern matching, bounded loops or recursion with termination arguments, Result/Option/Except, explicit effects, specifications, and ghost variables. Initially exclude or restrict any, eval, prototype mutation, dynamic property injection, reflection-heavy code, implicit undefined/null, arbitrary floating-point reasoning, arbitrary async/event-loop semantics, and arbitrary DOM manipulation.

Roadmap: Phase 1 verified pure TypeScript subset with pure functions, records, discriminated unions, arrays/lists, maps, integer arithmetic, requires/ensures, invariant, generated Lean, original JS/TS still running, Lean model generated, proof obligations generated, theorem manifest emitted. Phase 2 verified schemas and API contracts for HTTP handlers, RPC endpoints, database rows, environment variables, form data, and third-party API responses. Phase 3 verified React state with verified reducers, restricted hooks, verified component props, render-model specs, and impossible-state elimination. Phase 4 verified Bun server template with route handlers, request/response contracts, middleware invariants, typed error handling, validated config/env, and proof-aware test runner integration. Phase 5 proof-carrying npm registry convention with proof manifest format, CI verifier, GitHub Action, npm package metadata convention, proof badge, and consumer-side verification command. Phase 6 verified standard libraries for finite maps, non-empty arrays, sorted arrays, parsers, JSON codecs, URL/router matching, permission models, date/time intervals, money/decimal arithmetic, and state-machine helpers.

Biggest mistake to avoid: do not start with verifying all of React, all of Bun, full JavaScript, or arbitrary TypeScript-to-Lean compilation. Those are research programs, not an adoption strategy. Start with making one npm package able to ship checked Lean proofs for its public TypeScript API. That is small enough to build and large enough to matter.

Ranking by impact: proof-carrying npm packages for TypeScript is very high impact and medium feasibility; verified React reducers/state machines are high impact and high feasibility; verified Bun API/server contracts are high impact and medium-high feasibility; a TypeScript-to-Lean subset compiler is high impact and medium feasibility; Lean formal semantics of TSCore is high impact and medium feasibility; full TypeScript/JS semantics in Lean is very high impact but low feasibility; verified React reconciler and verified Bun runtime are lower-MVP priority because they are hard systems projects; a new TS+Lean language is high impact but weaker for ecosystem adoption than proof-carrying TypeScript.

Most impactful concrete contribution: contribute to or fork LemmaScript, but aim it at proof-carrying TypeScript packages for npm, with React and Bun integrations as first-class demos. First milestone: a Bun + React app where the domain model is ordinary TypeScript, reducers are verified, API contracts are verified, generated Lean contains no sorry, CI checks proofs, and the package publishes a proof manifest.

Bottom line: make verification package-native, gradual, Lean-checked, TypeScript-facing, focused first on domain/state/API boundaries, and honest about React, Bun, DOM, and third-party packages as contracted externals until specific pieces become worth verifying internally.

Question: To turn all code verifiable, is it better to verify existing code designed for humans or new code designed for agents and AI?

Answer: For the long-term goal of making all code verifiable, the better primary strategy is new code designed around verification and AI agents, not retrofitting arbitrary human-written code after the fact. But the practical strategy should be hybrid: long-term core is a new verification-native language/IR designed for agents and proof systems; near-term adoption uses wrappers, contracts, and gradual verification for existing TypeScript/React/Bun code; ecosystem bridge generates ordinary TS/JS/TSX plus Lean proof artifacts.

Position: the highest-leverage path is specification-first -> agent generates verified core -> Lean checks proof -> JS/TS/Bun/React output, rather than human-written arbitrary code -> reverse-engineer intent -> try to prove it correct.

Existing code often does not expose invariants. A verifier has to infer intent from implementation, but correctness is relative to a specification. Without a spec, the verifier can only prove shallow properties such as no crashes in some cases, no out-of-bounds access, no type mismatch, or maybe no forbidden side effect. It cannot prove business intent unless that meaning is written down. TypeScript makes this harder because its types are erased and annotations do not affect runtime behavior.

New agent-designed code is promising because AI agents can tolerate what humans dislike: verbose contracts, explicit state machines, ghost variables, proof hints, generated lemmas, normalized intermediate representations, large proof files, and repetitive refinement steps. Humans want code such as:

function transfer(from, to, amount) {
  from.balance -= amount
  to.balance += amount
}

A verifier wants:

requires amount > 0
requires balance(from) >= amount
requires from != to
ensures totalBalance(result) == totalBalance(input)
ensures balance(result, from) == balance(input, from) - amount
ensures balance(result, to) == balance(input, to) + amount

An agent can generate and maintain that burden. The primary artifact should not be pretty source code; it should be a checked semantic object.

Right architecture: human-facing syntax that is TS-like, React-compatible, and Bun-compatible; agent-facing verified IR that is small, explicit, proof-friendly, and normalized; and Lean backend that generates definitions, theorems, proof obligations, and checked proof artifacts. Stack: prompt/spec/design doc -> agent produces formal spec -> agent produces verified IR -> Lean checks proof -> compiler emits TS/JS/TSX -> Bun/React run executable output.

Verifying existing human code should not be the main plan because intent is missing, legacy code has implicit effects, existing ecosystems are not proof-shaped, and reverse verification does not scale cleanly. DARPA HACMS supports the lesson that verification is easier when systems are designed around formal methods from the start.

Existing code still matters. It should be handled as contracted external reality:

extern function stripeCharge(input: ChargeRequest): Promise<ChargeResult>
  assumes validChargeResultWhenSuccess
  assumes idempotencyKeyPreventsDoubleCharge

extern component ReactSelect(props: SelectProps): ReactNode
  assumes emitsOnlyValidOptions

The verifier does not need to prove Stripe, React, the browser, Bun, or every npm package immediately. It needs exact trust boundaries. Verified core: domain logic, state machines, reducers, permissions, parsers, codecs. Contract boundary: external libraries, runtime APIs, React, Bun, browser, database. Runtime checks: schemas, assertions, generated validators, fuzz tests, conformance tests.

Agent-native verified code should have explicit specifications for every exported function/component/module, explicit effects for IO/state/async/failure/randomness/time/network/database access, a small core language with no arbitrary JavaScript semantics in the proof layer, proof-carrying output, stable semantic IR, human-readable facade, and checker-in-the-loop proof repair.

Optimal strategy split: 70% new verification-native core for AI agents, 20% interop/contracts for existing TypeScript ecosystem, 10% direct verification of selected legacy libraries. The new core compounds. Existing code verification drives adoption.

For a React+Bun app, start by verifying domain models, state transitions, reducers, permissions, API contracts, database codecs, schema validators, payment/accounting logic, and routing invariants. Assume or contract React reconciler, DOM, Bun runtime, browser APIs, third-party npm packages, and database engine.

Example state machine:

machine OrderState {
  state Draft
  state Submitted
  state Paid
  state Cancelled
  transition submit: Draft -> Submitted
    requires cart.nonEmpty
  transition pay: Submitted -> Paid
    requires payment.authorized
  transition cancel: Draft | Submitted -> Cancelled
  invariant Paid implies payment.authorized
  invariant Cancelled implies not shipped
}

An agent can turn this into a Lean inductive state model, transition function, invariant theorem, generated reducer, generated TypeScript types, generated React hook, generated tests, and generated proof manifest. This is cleaner than taking a messy reducer and inferring the state machine after the fact.

Most important principle: do not make code the source of truth. Make specifications and proofs the source of truth. Executable code should be generated from or checked against the formal model. Bad future: AI writes code and humans hope it is right. Better future: AI writes spec + code + proof, Lean checks proof, and runtime code is generated or linked to that proof.

Bottom line: for making all code verifiable, do not bet primarily on old human-designed code. Bet on a new verification-native core language/IR designed for AI agents, checked by Lean, with TS/React/Bun compatibility at the edges and gradual contracts for existing code. Existing code should be wrapped, specified, tested, and gradually migrated. New critical code should be generated proof-first.

Product We Will Build

Bip: proof-carrying TypeScript packages.

One-line Product

Bip lets developers and agents write TypeScript-facing software that ships with Lean-checked contracts, so packages can prove important behavior at their public boundaries.

The sharper insight

The fastest path is not verifying all of JavaScript, React, Bun, or npm. It is making one package able to declare, check, and publish proofs for exported functions, reducers, schemas, components, and route contracts.

01

Human-facing syntax

Developers write near-ordinary TypeScript/TSX plus explicit contracts: requires, ensures, invariants, assumptions, and external boundaries.

02

Agent-facing IR

Agents transform specs into a small, normalized, verification-friendly intermediate language: TSCore.

03

Lean proof backend

Bip emits Lean definitions and theorem obligations, checks there is no unchecked proof hole, and records verified exports in a manifest.

04

Ecosystem output

The same package emits ordinary JavaScript, TypeScript declarations, runtime validators, and proof artifacts that Bun, React, and npm can consume without knowing Lean internals.

MVP

The first useful artifact is a verified React+Bun package template.

Build a package where a reducer, a domain model, and one API contract are written in TypeScript-like source, compiled to runtime TS/JS, modeled in Lean, checked in CI, and published with a proof manifest.

Pure functions, records, discriminated unions, Nat/Int/Bool/String, Result/Option, arrays/lists, contracts, generated Lean, generated .d.ts, and Bun-compatible output.

Not first

Full JavaScript semantics, React reconciler verification, Bun runtime verification, arbitrary async, DOM correctness, prototype mutation, any, eval, or unrestricted floating point.

Operating Principles

What the product should refuse to forget.

1

Verification must be package-native.

The npm package is the adoption unit. Proofs should travel with public APIs the way type declarations do.

2

Contracts beat inference.

The system should not pretend to infer business intent from arbitrary code. It should make intent explicit and check it.

3

Agents can carry proof ergonomics.

Humans need readable facades. Agents can manage verbose contracts, ghost state, proof hints, generated lemmas, and proof repair loops.

4

Externs are honest boundaries.

React, Bun, databases, browsers, Stripe, and third-party packages start as contracted externals with runtime checks and assumptions.

5

The source of truth is semantic.

Specs and checked proofs define the product's meaning. Executable code is generated from, or checked against, that meaning.
